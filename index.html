Systems Programming

  • Systems programming is unusually difficult compared to other forms of programming
  • OS development and design are the pinnacle or programming achievement, and the highest calling for any programmer
  • Systems programmers are inherently superior to other kinds of programmers
  • A competent systems programmer will naturally be gifted in all other forms of programming

I find these assumptions laughable. – Jay Osako

Systems programming has been divided from normal programming

  • Confined to C, since the end of LISP machines in the 1980s
  • Special tooling
  • Little code reuse
  • Poor debugging
  • Monoliths...
True, linux is monolithic, and I agree that microkernels are nicer... As has been noted (not only by me), the linux kernel is a miniscule part of a complete system: Full sources for linux currently runs to about 200kB compressed. And all of that source is portable, except for this tiny kernel that you can (provably: I did it) re-write totally from scratch in less than a year without having /any/ prior knowledge.– Linus Torvalds, 1992

Currently Linux has over 25 million lines of code...

... and Windows has 50 million.

System software is changing

Rust

  • zero-cost abstractions
  • guaranteed memory safety
  • threads without data races
  • type inference
  • minimal runtime

Go

  • From the Plan 9 operating system heritage
  • memory safety
  • strong distributed programming libraries

OCaml

  • Functional language
  • Full network stack implemented from TCP to SSL
  • Memory and type safe

Also many more languages

  • Haskell
  • Lua and LuaJIT
  • Elexir
  • JavaScript
  • Nim
  • D

Unikernels

Not reinventing the general purpose OS

No one is trying to remake Linux or Windows style monoliths

The model is just enough system software to run your code

Without any external dependencies, you have total control

Everything is open source, with liberal licenses.

A container with everything in it

A security hardened container

  • No large OS attack surface
  • Just what you need, no extra shell or other executables, so small attack surface
  • Can run inside virtual machine for sandboxing
  • Language guarantees, like type safety and memory safety
  • Can use additional sandboxing techniques: ASLR, NaCl etc
  • Whole system hardening
  • Ideal for embedded systems

And a distributed system of these containers

Distributed from the start

  • Single application unikernels mean that you have to build distributed systems
  • It is pretty difficult to build fat services
  • No fork or processes in a unikernel
  • Aim to reuse existing coordination code, and offer some new options...

Examples of unikernels

  • Mirage (OCaml)
  • HalVM (Haskell)
  • Ling (Erlang)
  • runtime.js (JavaScript)
  • ClickOS (for networking applications)

Rump kernels

  • A toolkit of reusable drivers for any unikernel to use, unmodified form NetBSD
  • Also can construct unikernels from existing code
  • Node.js, Python, PHP, Erlang
  • http://rumpkernel.org/

Deployment

Unikernels are at the stage that Linux containers were three years ago before Docker

  • Few users
  • Hard to build
  • Hard to ship
  • Hard to run

Clearly this needs to be fixed for widespread use...

"Wouldn't it be cool if we could deploy and manage unikernels with Docker?" Yes, yes it would! #dockercon pic.twitter.com/ZjJWkAuEQm

Deployment of unikernels via Docker

  • Prototype at Dockercon EU
  • Deployment with Docker, running under KVM for isolation
  • Rough edges but will integrate into Docker tools via pluggable drivers
  • Will give a drop-in deployment story

Common config

  • Mirage community developing configuration tool called Functoria
  • Aim to provide a common tooling for configuration of unikernels
  • Standard metadata
  • Integrate with Open Containers Specification?

Unikernel.org

  • Common community to share tooling, code and tests
  • Working on ways to reuse existing code across languages
  • Working on standard configuration and other layers
  • Take the learnings from Mirage and apply more broadly.
